Hi [Manager's Name], I wanted to share some feedback from the last release because I think there are a few process improvements that could help the team.

For this feature, I initially worked on four separate JIRAs. As development progressed, the implementation approach changed, the work was later consolidated, and the audit history implementation was also modified. Since we didn't have a detailed technical discussion on these changes before development started, a significant amount of rework was required. I worked additional hours to complete those changes and moved the code to QA around a week before the release.

However, the main QA validation happened very close to the release date, and the bugs were identified only at the last moment. That left very little time for fixes, regression testing, and validation before deployment.

I completely understand that I am responsible for the quality of the code I deliver, and I take that responsibility seriously. At the same time, I feel that expecting a developer to deliver a 100% bug-free feature without sufficient QA feedback before release isn't realistic. Human errors can happen, which is exactly why the development and QA processes complement each other. Quality is ultimately a shared responsibility across the team.

Going forward, I think we could reduce last-minute issues by:

Finalizing the technical approach before development starts as much as possible.
Minimizing requirement or implementation changes once development is in progress.
Starting QA validation earlier so there is enough time for bug fixing and regression before release.

I'm sharing this because I'd like to improve both my own delivery and the overall team process. I believe these changes would help us deliver more stable releases with less last-minute pressure.
